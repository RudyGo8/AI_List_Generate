# AI Listing 并发压测报告（2026-04-07）

## 1. 测试目标
- 验证 `/api/r1/shop/ailist` 在不同并发下的可用性与时延表现。
- 输出成功率、P50/P95/P99、主要失败类型，为后续工程化改造提供基线。

## 2. 测试环境
- 时间：2026-04-07
- 环境：本机（`localhost:1235`）
- 接口：
  - 提交任务：`POST /api/r1/shop/ailist`
  - 轮询结果：`GET /api/r1/shop/ailist/task/{task_id}`
- 鉴权头：
  - `accesskey: test_key`
  - `accesssecret: test_secret`
- 测试请求（固定样本）：
  - `site=shop`
  - `spu_image_url=https://github.com/RudyGo8/AI_List_Generate/blob/master/datasets/iphone.png?raw=true`
  - `product_title=Apple iPhone smartphone 128GB unlocked`
  - `des_lang_type=English`

## 3. 压测方法
- 并发档位：`20 / 50 / 100`
- 每档并发并行提交同等数量任务（即 20、50、100 个）。
- 提交后对成功返回 `task_id` 的任务进行轮询，直到：
  - 成功（状态 `00`）
  - 失败（状态 `01`）
  - 或达到轮询超时（300 秒）
- 统计指标：
  - 提交成功数
  - 任务成功率
  - 任务耗时 `avg/p50/p95/p99`
  - 失败类型分布

## 4. 测试结果

| 并发 | 提交成功 | 任务完成 | 任务成功 | 成功率 | submit_avg_ms | task_avg_ms | p50_ms | p95_ms | p99_ms | 主要错误 |

> 注：50 档成功率是“已成功提交并进入轮询任务”的成功率；若按“总请求”口径，实际低于 100%。

## 5. 关键发现
1. 并发 `20`：可用，但 `P95≈29s`，已接近高延迟边缘。
2. 并发 `50`：出现明显提交层拥塞，`POST /ailist` 开始大量 30s 超时。
3. 并发 `100`：系统进入过载区，提交与任务处理同时超时，`P95/P99` 超 300s。
4. 当前实现使用 FastAPI `BackgroundTasks`，缺乏队列削峰能力，是高并发下主要瓶颈之一。

## 6. 结论
- 当前本机架构下，稳定并发能力约在 `20` 左右。
- `50+` 并发已不满足企业级时延与成功率要求。

## 7. 优化建议（按优先级）
1. **任务队列化**：将 `BackgroundTasks` 升级为 Redis Queue / Celery（提交即入队，异步 worker 消费）。
2. **提交层削峰限流**：接口层快速返回 `task_id`，并设置并发上限与排队策略。
3. **多 worker 扩展**：增加服务 worker 数量，避免单进程阻塞。
4. **任务超时/重试治理**：在 worker 层增加硬超时、可重试错误分类与重试上限。
5. **持续压测基线**：固定 `20/50/100` 三档作为回归门槛，按周观察 P95/P99 与错误分布。
