# AI Listing Generate 数据库表架构文档（企业级）

## 1. 概览
当前项目数据库为 MySQL，核心按 4 个域组织：
1. 系统配置域：模型配置、提示词、客户端鉴权
2. 商品任务域：源数据、任务、目标数据、回调通知
3. 类目域：站点类目与属性配置
4. 批处理/标签域：批次信息、标签配置

## 2. 核心表清单
### 2.1 配置域
1. `db_sys_conf`：系统配置键值表
2. `db_sys_ai_prompt`：AI 提示词模板表
3. `db_sys_client_conf`：客户端鉴权配置表
4. `db_ai_model_route`：按任务类型/场景的模型路由表（代码已依赖）

### 2.2 商品任务域
1. `db_product_src_detail`：原始商品数据
2. `db_product_task_detail`：任务状态与执行信息
3. `db_product_des_detail`：生成后的商品数据
4. `db_notice_detail`：回调通知记录

### 2.3 类目域
1. `db_category`：站点类目表（embedding/hybrid 匹配）
2. `db_product_category_conf`：类目配置
3. `db_product_category_attribute_conf`：类目属性配置

### 2.4 批处理/标签域
1. `db_sys_batch_info`：批次信息
2. `db_product_tag_conf`：标签配置

## 3. 关键关系（逻辑外键）
1. `db_product_task_detail.product_src_id -> db_product_src_detail.id`
2. `db_product_task_detail.product_des_id -> db_product_des_detail.id`
3. `db_product_des_detail.product_src_id -> db_product_src_detail.id`
4. `db_notice_detail.biz_id` 与业务表主键关联（按 `task_type` 解释）
5. 多表共享字段 `gid` 作为客户端维度隔离键

## 4. 关键字段与索引（摘要）
### 4.1 `db_product_src_detail`
1. 主键：`id`
2. 关键字段：`gid`, `spu_image_url`, `product_title`, `category_name`, `attributes`
3. 索引：`idx_spd_gid`, `idx_spd_create_time`, `idx_spd_update_time`, `idx_spd_product_title`

### 4.2 `db_product_task_detail`
1. 主键：`id`
2. 关键字段：`status`, `product_src_id`, `product_des_id`, `site`, `platform_id`, `des_lang_type`, `usage`
3. 状态枚举：`00` 成功，`01` 失败，`02` 就绪，`03` 处理中
4. 索引：`idx_ptd_status`, `idx_ptd_gid`, `idx_ptd_platform_id`, `idx_ptd_product_src_id`

### 4.3 `db_product_des_detail`
1. 主键：`id`
2. 关键字段：`product_title`, `product_desc`, `category_name`, `category_id`, `attr_value_list`
3. 索引：`idx_dpd_gid`, `idx_dpd_src_product_src_id`, `idx_dpd_src_platform_id_site`

### 4.4 `db_notice_detail`
1. 主键：`id`
2. 关键字段：`task_type`, `biz_id`, `notice_url`, `notice_counts`, `next_notice_time`, `status`
3. 索引：`idx_nd_status`, `idx_nd_next_notice_time`, `idx_nd_task_type_biz_id`

### 4.5 `db_sys_conf`
1. 主键：`id`
2. 关键字段：`key`, `value`, `enable`
3. 唯一索引：`uniq_sc_key_enable`

### 4.6 `db_sys_ai_prompt`
1. 主键：`id`
2. 关键字段：`prompt_key`, `prompt_value`, `enable`
3. 唯一索引：`uniq_sap_prompt_key_enable`

### 4.7 `db_sys_client_conf`
1. 主键：`id`
2. 关键字段：`client_name`, `api_key`, `api_secret`, `enable`
3. 索引/约束：建议补充唯一约束（当前 ORM 未完整体现）

### 4.8 `db_category`
1. 主键：`id`
2. 关键字段：`site`, `category_path`, `category_id`, `enable`
3. 唯一索引：`site + category_id`

## 5. 现状差异与风险
1. `create_tables.sql` 中未包含 `db_ai_model_route` 建表语句，但代码与初始化数据已使用该表。
2. `db_category` DDL 在 SQL 文件末尾存在注释和语句拼接风险，需验证初始化是否稳定执行。
3. `db_product_task_detail.usage` 为 `VARCHAR(128)`，与实际 usage JSON 存储需求冲突。
4. SQL 与 schema/ORM 存在字段类型不一致风险（例如 `category_id`）。
5. 目前大多是逻辑外键，无数据库层 FK 约束，存在数据一致性风险。

## 6. 企业级改造要求
1. 建立数据库迁移体系（建议 Alembic）。
2. 补齐缺失表和约束，统一字段类型与字符集。
3. `usage`、`custom_data`、`notice_content` 建议升级为 `TEXT/JSON`。
4. 建立归档策略：任务与通知历史表分离，控制主表体积。
5. 建立索引治理：按查询路径做慢 SQL 与索引命中率监控。

## 7. 建议补丁（DDL层）
1. 在 `create_tables.sql` 新增 `db_ai_model_route` 建表。
2. 修复 `db_category` 注释与建表语句拼接问题。
3. 调整 `db_product_task_detail.usage` 字段长度或类型。
4. 对关键查询路径补充组合索引（例如 `gid + status + create_time`）。
